# ProCount — מסמך אפיון

**תאריך:** 2026-06-18
**מטרה:** אפליקציית אינטרנט אישית (משתמש יחיד) למעקב יומי אחר צריכת **חלבון** ו**קלוריות**, בשימוש בעיקר מהאייפון.
**עקרון מנחה:** יעיל, נוח, לא מסובך. כל פיצ'ר שלא משרת את זה — בחוץ.

---

## 1. סקירה

מסך יומי שמדגיש כמה חלבון אכלת מול יעד, עם הוספה מהירה דרך רשימת מאכלים אישית, הזנה ידנית, או צילום שמנותח ע״י Claude. קלוריות נספרות לצד החלבון בלי יעד. היסטוריה מלאה נשמרת ומוצגת כגרפים ומגמות.

---

## 2. החלטות מפתח (ננעלו בסיעור המוחות)

| נושא | החלטה |
|---|---|
| הזנת מזון | רשימה אישית (לחיצה=הוספה) + הזנה ידנית + צילום→ניתוח AI |
| יעד | חלבון בלבד; קלוריות = מספר רץ ללא יעד |
| היסטוריה | גרפים ומגמות (שבועי/חודשי, ממוצע, streak) |
| ארכיטקטורה | PWA סטטי + Supabase (DB + Auth + Edge Function ל-AI) |
| פלטפורמה | אפליקציית אינטרנט מותקנת למסך הבית של האייפון (PWA) |

---

## 3. ארכיטקטורה

- **Frontend:** אפליקציית דף-יחיד (SPA), נארזת כ-PWA (manifest + service worker) כדי שתתקין למסך הבית ותיטען מיידית. השכבה הוויזואלית מעוצבת בנפרד (claude design) — מסמך זה מגדיר מבנה והתנהגות בלבד.
- **Supabase:**
  - **Postgres** — הנתונים (טבלאות בסעיף 5).
  - **Auth** — משתמש יחיד (סעיף 8).
  - **Edge Function** `analyze-food-photo` — מקבלת תמונה, קוראת ל-Claude vision, מחזירה JSON. מחזיקה את מפתח ה-API בצד-שרת (סעיף 7).
- **תקשורת:** ה-Frontend מדבר ישירות מול Supabase (קריאה/כתיבה דרך supabase-js) ומול ה-Edge Function (לצילום בלבד).
- **אירוח:** ה-Frontend הסטטי על כל אירוח סטטי (Vercel/Netlify/Cloudflare Pages — חינם). אין שרת משלנו לתחזק.

---

## 4. מודל נתונים

כל הטבלאות כוללות `user_id uuid` עם `default auth.uid()`, ו-**RLS**: `user_id = auth.uid()` לקריאה ולכתיבה. כך אף אחד לא ניגש לנתונים של אחר (וגם שכבת הגנה על האפליקציה).

**`foods`** — רשימת מאכלים אישית (תבניות לשימוש חוזר):
| שדה | טיפוס | הערה |
|---|---|---|
| id | uuid (PK) | |
| user_id | uuid | default auth.uid() |
| name | text | |
| protein_g | numeric | ≥ 0 |
| calories | numeric | ≥ 0 |
| default_qty | numeric | ברירת מחדל 1 (כפלן כמות) |
| created_at | timestamptz | default now() |

**`entries`** — כל רישום צריכה בפועל:
| שדה | טיפוס | הערה |
|---|---|---|
| id | uuid (PK) | |
| user_id | uuid | default auth.uid() |
| eaten_on | date | התאריך שאליו הרישום משויך |
| name | text | |
| protein_g | numeric | ≥ 0 |
| calories | numeric | ≥ 0 |
| source | text | 'manual' \| 'saved' \| 'ai' |
| food_id | uuid (FK→foods, nullable) | אם הגיע מרשימה |
| created_at | timestamptz | default now() |

**`profile`** — שורה אחת למשתמש:
| שדה | טיפוס | הערה |
|---|---|---|
| user_id | uuid (PK) | |
| protein_goal_g | numeric | היעד היומי לחלבון |

**גבול היום:** `eaten_on` הוא `date`. הלקוח מחשב את "היום" באזור הזמן המקומי ושולח אותו. תצוגת "היום" = `entries` עם `eaten_on = התאריך המקומי הנוכחי`. ניתן לערוך תאריך ברישום (השלמה אחורה).

## 5. מסכים וזרימות (פונקציונלי)

- **היום (בית):**
  - חלבון בגדול: `X / יעד g` + "נשאר Y" (טבעת/בר).
  - קלוריות כמספר רץ בצד (ללא יעד).
  - רשימת רישומי היום → עריכה/מחיקה.
  - כפתור ראשי **+ הוסף**.
- **הוספה** (המסך הכי בשימוש — חייב מהיר):
  1. **מהיר:** רשת המאכלים השמורים → לחיצה = הוספה מיידית עם כפלן כמות (×1 / ×2 / …). יוצר `entry` עם `source='saved'`.
  2. **ידני:** טופס (חלבון + קלוריות, שם אופציונלי) → הוספה. צ'קבוקס "שמור למאכלים שלי" (יוצר גם `food`). `source='manual'`.
  3. **צילום:** צלם/בחר → המתנה → תוצאת AI **ניתנת לעריכה** → אישור. `source='ai'`. צ'קבוקס "שמור למאכלים שלי".
- **מגמות:**
  - גרף חלבון יומי (7 / 30 יום) עם קו יעד.
  - ממוצע חלבון יומי בטווח.
  - גרף קלוריות יומי (משני).
  - **streak:** מספר הימים הרצופים (עד היום/אתמול) שבהם סך החלבון ≥ היעד.
- **מאכלים שלי:** רשימה לניהול — הוסף / ערוך / מחק.
- **הגדרות:** יעד חלבון, התחברות/התנתקות.
- **ניווט:** 3 טאבים תחתונים נוחים לאגודל — **היום · מגמות · מאכלים**. הגדרות בתוך טאב (אייקון פינתי).

הגרפים וה-streak מחושבים בצד-לקוח מתוך `entries` + היעד (אגרגציה לפי `eaten_on`).

## 6. זרימת הצילום → AI

1. הלקוח **מקטין/דוחס** את התמונה (לדוגמה צלע ארוכה ~1024px, JPEG) לפני שליחה — מצמצם טוקנים ומשקל.
2. שולח את התמונה (base64) ל-Edge Function `analyze-food-photo` (מאומתת — סעיף 8).
3. ה-Edge Function קוראת ל-Claude (Deno/TypeScript) עם:
   - **תוכן הודעה:** בלוק `image` (base64) + בלוק `text` עם הוראה לאמוד מאכל, קלוריות וחלבון.
   - **פלט מובנה** דרך `output_config.format` עם JSON schema:
     ```
     { name: string, calories: number, protein_g: number,
       confidence: "low" | "medium" | "high", note: string }
     ```
     (`note` = הנחה קצרה שה-AI עשה, למשל "הנחתי מנה 150 גרם" — עוזר למשתמש לתקן.)
   - **מודל:** `claude-sonnet-4-6` (נבחר — איזון מצוין של דיוק ראייה, מהירות ועלות). קבוע יחיד בקוד (`MODEL_ID`) שקל להחליף בעתיד.
   - **ללא extended thinking, ללא streaming, `max_tokens` קטן (~1024)** — האומדן גס וקצר; פשטות וזריזות.
   - מפתח `ANTHROPIC_API_KEY` נשמר כ-Supabase secret, לעולם לא בצד-לקוח.
4. התוצאה חוזרת ללקוח ומוצגת **בשדות ניתנים לעריכה** — המשתמש מאשר/מתקן לפני שמירה (אומדן AI הוא בערך).
5. כשל / חוסר ודאות → נפילה להזנה ידנית עם מה שכן זוהה ממולא מראש.

---

## 7. אימות והרשאות

- **Supabase Auth**, התחברות חד-פעמית במייל (magic-link, ללא סיסמה). הסשן נשמר במכשיר → כמעט לא תתחבר שוב.
- **RLS** על כל הטבלאות לפי `auth.uid()`.
- ה-Edge Function דורשת JWT תקף (משתמש מחובר) → רק אתה יכול להפעיל אותה. מונע ניצול והוצאת קרדיט AI.
- **תקרה רכה:** מקסימום **6 קריאות AI ביממה** למשתמש (בלם עלות). מימוש: ספירת `entries` עם `source='ai'` בתאריך הנוכחי לפני הקריאה ל-Edge Function (או מונה ב-`profile`); מעל המכסה — חוסמים בנימוס ומפנים להזנה ידנית.

---

## 8. טיפול בשגיאות וקצוות

- אומדן AI שגוי → תמיד ניתן לעריכה לפני שמירה; מוצגת רמת `confidence` ושדה `note`.
- מפתח API → רק ב-Edge Function, אף פעם בלקוח.
- ולידציה: `protein_g`, `calories` — מספרים אי-שליליים; `name` אופציונלי בהזנה ידנית.
- אופליין: ה-PWA נטען מהמטמון (app shell), אבל רישום/קריאת נתונים דורשים חיבור. *(הפשטה מכוונת — תור-אופליין הוא פאזה 2 אם בכלל יידרש.)*
- סשן פג → התחברות מחדש (magic-link).
